    第21回飛行ロボコン画像修正作業 - インシデント分析報告書¶

    作成日: 2025/09/16
    作成者: Claude Code
    重要度: 🔴 Critical

    📋 エグゼクティブサマリー¶

    自動画像参照置換処理において、以下の3つの重大な問題が発生： 1. 無関係画像の誤挿入: missions系画像がfield画像位置に配置 2. 画像の重複挿入: 同一画像が3回連続で挿入
    3. キャプションとの不整合: 画像とキャプションの対応関係が崩壊

    🔴 発生した問題の詳細¶

    問題1: 無関係画像の誤挿入¶

    症状: - 176-177行目にmissions-自動８の字飛行ミッション図解_21.png等の無関係画像が挿入 - 本来挿入すべきfield系画像の前に不適切な画像が配置

    影響範囲: - 一般部門フィールドレイアウト図セクション全体

    問題2: 画像の重複挿入¶

    症状: - equipment-滑走路_21.pngが3回連続挿入（191, 193, 195行） - マルチコプター部門画像が3セット重複（199-206行）

    影響範囲: - 滑走路セクション、マルチコプター部門セクション

    問題3: キャプション位置の不整合¶

    症状: - 画像とキャプション「図　〇〇」の対応関係が崩壊 - キャプションが画像から離れた位置に配置

    影響範囲: - 文書全体の画像参照部分

    🔍 根本原因分析（RCA）¶

    原因1: Module B - 削除対象画像の処理ロジック不備¶

    コード問題箇所: module_b_replace_existing.py 81-88行目 

    if skip_line:
    # この行をスキップ
    i += 1
    # 次の行がキャプションの場合もスキップ
    if i < len(lines) and lines[i].strip().startswith('図'):
        i += 1
    continue

    問題点: - image1-3削除時に、同じ行にあった他のimage参照も巻き込んで処理 - 複数画像が1行にある場合の処理が不適切

    原因2: Module C - マーカー位置特定の不正確さ¶

    コード問題箇所: module_c_add_new_images.py 72-73行目 

    for i, line in enumerate(lines):
    if marker in line:
        marker_found = True

    問題点: - 部分一致で判定したため、誤った位置を検出 - 既存画像との位置関係を考慮していない - 同じループ内で複数回処理され、重複挿入が発生

    原因3: テスト戦略の不備¶

    テスト不足箇所: 1. 数値検証のみ: 画像数のカウントのみで、配置の妥当性未検証 2. 視覚的確認なし: MkDocsプレビューでの確認を省略 3. 差分確認なし: 変更前後の詳細な差分分析未実施

    📊 レビュープロセスで見逃した理由¶

    1. 過度の自動化信頼¶

    • モジュールテストの「SUCCESS」表示を過信
    • 数値的な成功（34個の画像参照）に満足

    2. コンテキスト理解不足¶

    • image1-3の元の位置と内容を正確に把握していなかった
    • 文書構造と画像の意味的関係を軽視

    3. エンドツーエンドテストの欠如¶

    • 単体テストは実施したが、統合テストが不十分
    • 実際のレンダリング結果を確認しなかった

    🛡️ 再発防止策¶

    1. コード修正要件¶

    Module B改修¶

    # 改修案: 削除対象を正確に特定
for img_id in matches:
    if img_id in IMAGE_MAPPING:
        if IMAGE_MAPPING[img_id] is None:
            # 該当箇所のみを削除（行全体ではなく）
            new_line = new_line.replace(f'![][{img_id}]', '')

    Module C改修¶

    # 改修案: 完全一致と重複防止
already_processed = set()
for config in NEW_IMAGES:
    if config['marker'] in already_processed:
        continue
    # 完全一致でマーカーを検索
    if line.strip() == config['marker']:
        # 処理実行
        already_processed.add(config['marker'])

    2. テスト強化策¶

    必須テストチェックリスト¶

    • [ ] 配置検証: 各画像が正しいセクションに配置されているか
    • [ ] 重複検証: 同一画像の不要な重複がないか
    • [ ] 整合性検証: 画像とキャプションが対応しているか
    • [ ] プレビュー検証: MkDocsで実際の表示を確認
    • [ ] 差分検証: git diffで変更内容を詳細確認

    3. プロセス改善¶

    段階的実行プロトコル¶

    1. Dry-run実行: 変更内容を事前にシミュレート
    2. 部分適用: セクションごとに段階的に適用
    3. 即時確認: 各段階でプレビュー確認
    4. ロールバック準備: 各段階でのバックアップ

    📝 教訓（Lessons Learned）¶

    技術的教訓¶

    1. 複雑な文字列処理には正規表現の精密な設計が必要
    2. 配列のインデックス操作は副作用に注意
    3. マーカーベースの処理は完全一致を原則とする

    プロセス的教訓¶

    1. 自動化は人間の確認を代替しない
    2. 成功メッセージは必要条件であって十分条件ではない
    3. 視覚的確認は最終品質保証に不可欠

    文化的教訓¶

    1. 「最高レベルの思考能力」は慎重さと徹底性を意味する
    2. 手戻りのコストは初期の慎重さのコストを上回る
    3. 部分的成功を完全な成功と誤認しない
